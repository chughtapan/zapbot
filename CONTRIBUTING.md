# Contributing

## Setup

```bash
git clone https://github.com/<your-org>/zapbot.git
cd zapbot
./install.sh    # Installs ngrok, bun, plannotator, agent-orchestrator. Creates .env and test repo.
./start.sh      # Starts webhook-bridge + AO + ngrok tunnel
```

Prerequisites: node 20+, git 2.25+, bun, gh CLI (authenticated), tmux, claude, ngrok (with auth token).

## Project structure

```
zapbot/
├── bin/
│   ├── webhook-bridge.ts      # Bun HTTP server (port 3000)
│   ├── zapbot-publish.sh      # Plan → GitHub issue publisher

├── .claude/skills/
│   └── zapbot-publish/
│       └── SKILL.md            # Claude Code skill (thin wrapper)
├── test/
│   └── e2e-smoke.sh           # E2E tests (14 tests)
├── install.sh                  # Tool installer + repo setup
├── start.sh                    # One-command startup
├── agent-orchestrator.yaml     # AO config
├── .agent-rules.md             # Agent implementation instructions
├── CLAUDE.md                   # Project instructions for Claude Code
├── .env                        # Secrets (gitignored, generated by install.sh)
└── .gitignore
```

## Running tests

```bash
./test/e2e-smoke.sh
```

Tests require `gh` CLI authenticated and the `zapbot-test` repo to exist. The bridge must be running for bridge-specific tests (Test 6). Tests create and clean up real GitHub issues.

## Development workflow

1. Edit code
2. Restart the bridge: `pkill -f "bun.*webhook-bridge" && bun bin/webhook-bridge.ts &`
3. Run tests: `./test/e2e-smoke.sh`
4. Test the full flow: create a plan, publish, add label, verify agent spawns

## Key files

| File | What it does | When to modify |
|------|-------------|----------------|
| `bin/webhook-bridge.ts` | Webhook routing, spawn triggers, callbacks | Adding new webhook handlers or callback types |
| `bin/zapbot-publish.sh` | Issue creation/update logic | Changing how plans are published |
| `bin/share-link.ts` | Plannotator URL generation | Changing share link format |
| `agent-orchestrator.yaml` | AO config (port, repo, reactions) | Adding repos, changing agent behavior |
| `.agent-rules.md` | Instructions injected into the agent's prompt | Changing how agents implement plans |
| `test/e2e-smoke.sh` | E2E test suite | Adding tests for new features |

## Adding a new repo

Add an entry to `agent-orchestrator.yaml` under `projects:`:

```yaml
projects:
  new-project:
    repo: your-org/new-repo
    path: ~/new-repo
    defaultBranch: main
    sessionPrefix: np
    agentRulesFile: .agent-rules.md
```

Then create the `zapbot-plan` and `plan-approved` labels on that repo:

```bash
gh label create "zapbot-plan" --repo your-org/new-repo --color "0E8A16" --force
gh label create "plan-approved" --repo your-org/new-repo --color "1D76DB" --force
```
